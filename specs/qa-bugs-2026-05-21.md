## BUG-005 — Dashboard date format

**Reported:** "JUEVES 21 DE MAYO" — spec §4.3 requires "21 may 2026"

**Root cause:**  
`formatDateKicker` in `apps/web/src/pages/Dashboard/helpers.ts:20-29` returns full day name + full month name: `"Jueves 21 de mayo · 3:45 PM"`. The kicker element in `PageHeader.tsx:17` has `className="... uppercase ..."`, which CSS-transforms the whole string to uppercase → "JUEVES 21 DE MAYO".

**Fix:**  
Rewrite `formatDateKicker` to return `"21 may 2026 · 3:45 PM"` format (numeric day + 3-char month abbreviation + 4-digit year). Remove the `uppercase` CSS class from the kicker `<div>` since the spec shows lowercase abbreviated format.

**Files:**

- `apps/web/src/pages/Dashboard/helpers.ts` — rewrite `formatDateKicker`
- `apps/web/src/pages/Dashboard/PageHeader.tsx:17` — remove `uppercase` class

## BUG-011 — Audit log AM/PM dots

**Reported:** Timestamps render "p. m." / "a. m." — spec §4.3 requires "PM"/"AM"

**Root cause:**  
`formatTs` in `apps/web/src/pages/settings/AuditLog.tsx:76` calls `d.toLocaleString('es-DO', { hour: 'numeric', minute: '2-digit', hour12: true })`. The `es-DO` locale formats 12-hour markers as "a. m." / "p. m." (with dots and spaces) per CLDR Spanish rules.

**Fix:**  
Replace `toLocaleString` call with manual formatting — extract hours/minutes, compute AM/PM manually same pattern as `formatDateKicker`:

```ts
const hours = d.getHours()
const minutes = d.getMinutes().toString().padStart(2, '0')
const ampm = hours >= 12 ? 'PM' : 'AM'
const h12 = hours % 12 || 12
const time = `${h12}:${minutes} ${ampm}`
```

**Files:**

- `apps/web/src/pages/settings/AuditLog.tsx:71-78` — replace `formatTs` body

## BUG-038 — Audit entity names truncated

**Reported:** Shows "Protocol303d5b24…" instead of "Retest Editor Protocol"

**Root cause:**  
`extractEntityName` in `apps/api/src/common/interceptors/audit-log.interceptor.ts:21-26` extracts entity name from the HTTP response body by looking for `response.name ?? response.fullName ?? response.title`. This works for CREATE/UPDATE operations (response includes the entity). For **DELETE** operations, the response body is usually empty or just `{ id }`, so `entityName` is undefined. The frontend then falls back to showing `entityType + entityId.slice(0,8)` (`apps/web/src/pages/settings/AuditLog.tsx:514-521`).

The inconsistency ("some creations show full name, others UUID-prefixed") occurs because some entities don't match the field name check — e.g., a protocol uses `title`, which IS checked, but the ProtocolVersion create response may use a different shape.

**Fix:**  
Two-part fix:

1. **Backend:** For DELETE operations, fetch the entity name before deletion and store it in the audit context metadata. Each repository's `delete` method should call `auditLog.setMetadata({ entityName })` with the entity's name before deletion. Alternatively, enrich the audit interceptor to handle DELETE by reading the entity name from a pre-deletion context set by the service.
2. **Frontend (fallback):** For cases where `metadata.entityName` is missing, display `entityType` in a friendlier label (e.g. "Protocolo" not "Protocol") rather than the raw UUID. This covers any gap until backend is fixed.

**Files:**

- `apps/api/src/common/interceptors/audit-log.interceptor.ts` — handle DELETE entity name
- `apps/api/src/common/audit-log/audit-context.store.ts` — potentially store pre-delete name
- `apps/web/src/pages/settings/AuditLog.tsx:514-521` — improve fallback display (friendly entity type label)

## BUG-017 — Location form button disabled, no validation feedback

**Reported:** "Crear ubicación" disabled when name empty; no inline error messages or required markers on submit attempt

- Add required marker (`*`) to the Name `<Field>` (already supported via `required` prop on `Field`).
- Either keep button disabled (acceptable) OR enable button and show errors on click — the QA expects visible validation feedback on submit attempt.

**Files:**

- `apps/web/src/pages/settings/Locations.tsx` — add submitted state + error display

## BUG-018 — Location form labels missing `htmlFor`

**Reported:** 9 `<label>` elements, 0 have `for` attribute; inputs lack programmatic label association

**Root cause:**  
The `Field` component in `apps/web/src/components/ui/Input.tsx` renders a `<label>` but never accepts an `htmlFor`/`id` prop — it has no way to associate the label with its child input. All usages of `<Field label="...">` across the app are affected.

**Fix:**

- Add optional `id` prop to `FieldProps`.
- Pass `htmlFor={id}` to the `<label>` element.
- In `Locations.tsx`, add explicit `id` to each `<Field>` and matching `id` to each `<Input>` (e.g. `id="location-name"`, `id="location-city"`, etc.).

**Files:**

- `apps/web/src/components/ui/Input.tsx` — add `id` prop to `Field`, set `htmlFor`
- `apps/web/src/pages/settings/Locations.tsx` — pass `id` to each `Field` + matching `id` on `Input`

## BUG-036 — Dashboard mobile overflow + sidebar not collapsing

**Reported:** Dashboard scrollWidth=462 > clientWidth=375 at 375px; sidebar stays 240px on all pages

**Root cause (two issues):**

1. **Dashboard grid:** `apps/web/src/pages/Dashboard/index.tsx:101` uses `grid grid-cols-4 gap-5` with no responsive breakpoint. At 375px, the 4-column grid has no room to fit — forces horizontal scroll.

2. **Sidebar width on mobile:** `apps/web/src/components/layout/Sidebar.tsx:132-137` applies `w-sidebar` (240px) unconditionally. On mobile, the sidebar hides via `-translate-x-full` but the layout still reserves the sidebar width with `lg:ml-sidebar` on the content area. At < `lg` breakpoint this should work, but if any content has a fixed width wider than the viewport minus sidebar, horizontal overflow occurs. Additionally, the sidebar doesn't collapse at all (no hamburger/toggle mechanism for mobile is visible per the QA report).

**Fix:**

1. Add responsive grid classes to the dashboard stats grid: `grid grid-cols-2 md:grid-cols-4`.
2. Audit all `grid-cols-N` and fixed-width containers on the dashboard for mobile responsiveness.
3. Add a mobile menu toggle (hamburger button in the topbar) that controls sidebar open/closed state, and ensure the layout does NOT reserve sidebar margin on mobile (`ml-0` at `< lg`, `lg:ml-sidebar`).

**Files:**

- `apps/web/src/pages/Dashboard/index.tsx` — add responsive grid breakpoints
- `apps/web/src/components/layout/Sidebar.tsx` — verify translate behavior
- `apps/web/src/components/layout/AppLayout.tsx` — verify margin at mobile breakpoints

## BUG-008 — Fresh account lands at `/bienvenido` gate blocking dashboard

**Reported:** New account can't reach dashboard; onboarding frames system templates as opt-in

**Root cause:**  
`OnboardingGate.tsx:14` checks `user.tenantSeededAt !== null` — if null, the user stays on the `/bienvenido` route and cannot reach any other route. The gate is correct by design (onboarding must complete before dashboard access), but the onboarding flow itself presents the seeded system templates as an optional selection ("opt-in") when per §12.6/§15.1 they should be automatically available without user action.

**Fix:**  
Two possible approaches:

- **Option A (preferred):** Auto-complete tenant seeding on account creation without user action on system templates. The `/bienvenido` screen should only collect essential info (name/specialty) and then automatically seed all system templates in the background. Remove the "choose templates" step from onboarding.
- **Option B:** Keep the opt-in step but change framing to "these templates are already active — customize later" with a "Continue" CTA that seeds and redirects.

Verify the `tenant-seeding.service.ts` seeds system templates automatically and that `tenantSeededAt` is set immediately on account creation (not deferred to user action).

**Files:**

- `apps/api/src/modules/tenant-seeding/tenant-seeding.service.ts` — verify auto-seed behavior
- `apps/web/src/pages/Onboarding/` — update onboarding flow to not gate on template selection

## BUG-022 — System templates show "Bloqueada por 1 tipo"

**Reported:** `/ajustes/plantillas` shows opaque "Bloqueada por 1 tipo" badge on system templates

**Root cause:**  
`protocol-templates.service.ts:17-18` sets `isLocked = blockingTypeIds.length > 0` — any template referenced by a protocol type is considered locked. This is intentional to prevent editing/deleting in-use templates. However, system/seeded templates (`isSeeded: true`) that are in use show "Bloqueada por 1 tipo" as their status badge in `Templates.tsx:112-118`, overriding the expected "Sistema" indicator. The badge is misleading — users see a warning status when the template is simply in use by the configured protocol type.

**Fix:**  
In `Templates.tsx`, distinguish between seeded and non-seeded locked templates:

```tsx
{
  t.isSeeded ? (
    <Badge variant="draft">Sistema</Badge>
  ) : t.isLocked ? (
    <Badge variant="review">{templatesStrings.listBlockedBy(t.blockingTypeIds?.length ?? 0)}</Badge>
  ) : (
    <Badge variant="active">Activa</Badge>
  )
}
```

Seeded templates should show "Sistema" (neutral/draft badge) regardless of lock state, since they can't be deleted by design.

**Files:**

- `apps/web/src/pages/settings/Templates.tsx:112-118` — add `isSeeded` check before `isLocked`
